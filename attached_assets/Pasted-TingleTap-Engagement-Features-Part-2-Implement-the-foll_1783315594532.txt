TingleTap Engagement Features – Part 2

Implement the following features while keeping Firebase usage extremely efficient.

Critical Requirements

Do NOT change:

- Existing UI theme
- Existing styling
- Existing moderation system
- Existing trust system
- Existing chat architecture (except where requested)
- Existing Firebase collections unless absolutely necessary
- Firestore/RTDB rules
- Existing animations
- Existing routing
- Existing chat functionality outside these requested features

The implementation must prioritize:

- Extremely low Firestore reads
- Extremely low Firestore writes
- Extremely low RTDB usage
- No duplicate listeners
- No polling
- No unnecessary document updates
- No unnecessary re-renders

Reuse existing components and user documents wherever possible.

Run "npm run build" after implementation.

1. Relationship Mark System

Allow users to privately mark another user using one of the following relationship types:

- Friend
- Best Friend
- Partner
- Family
- Favourite

This is a personal relationship marker.

Changing the relationship from one user must NOT affect the other user's profile automatically.

Only the user who creates the relationship should see it.

2. Homepage Chat Composer

Replace the current single-line message input with a modern premium message composer.

---

Multi-Line Composer

Requirements:

- Convert the current input into an auto-expanding multiline text area.
- Pressing Enter creates a new line.
- Messages are sent only by pressing the existing Send button.
- Enter must never send a message.
- Automatically grow downward while typing.
- Enable an internal vertical scrollbar only after reaching a reasonable maximum height.
- Preserve existing mentions, emoji picker, reply functionality, send button, and current chat features.
- Do not change the visual style.
- Keep desktop and mobile behaviour consistent.

---

Maximum Message Length

Implement a strict limit.

Requirements:

- Maximum 240 characters.
- Prevent typing beyond 240 characters.
- Prevent pasting beyond 240 characters.
- Validate again before sending.
- Never allow more than 240 characters to reach Firebase.
- Never display more than 240 characters inside chat.
- Show a premium toast/error notification if validation fails.

---

Live Character Counter

Display:

"0 / 240"

Requirements:

- Update live.
- Warning color after 220 characters.
- Stop additional typing at 240.
- Match existing TingleTap design.

4. Firebase Optimization

The implementation must remain lightweight.

Requirements:

- Reuse existing Firestore listeners.
- Reuse existing RTDB listeners.
- Never create duplicate listeners.
- Never poll Firestore.
- Never poll RTDB.
- Avoid collection scans.
- Cache frequently used values during the session.
- Batch writes wherever possible.
- Update only changed fields.
- Use merge updates.
- Keep RTDB traffic unchanged unless absolutely necessary.

Relationship updates should perform only a single Firestore update when changed.

The message composer improvements must not increase Firebase reads or writes.
